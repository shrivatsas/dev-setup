# C4 Guidance

Use C4 when the user needs visual architecture communication: who uses the system, what major software units exist, how they communicate, and how they deploy.

## Views

### L1 System Context

Show:

- Primary users and user groups.
- The system under discussion as one box.
- Adjacent products and external systems.
- Identity, observability, platform, and important third-party dependencies when they materially affect architecture.

Avoid:

- Internal components.
- Databases unless they are external/shared systems at this level.

### L2 Container

Show separately deployable/runnable units:

- Web apps, mobile apps, APIs, workers, schedulers.
- Databases, caches, queues, object stores.
- Identity providers and shared external services.
- Observability sinks.

Include protocol or relationship labels where useful: REST, GraphQL, events, queue, signed URL, JWT, S3 object access.

### L3 Component

Show major internal building blocks inside one container:

- Frontend modules, providers, service layer, query/data layer.
- Backend domains, API router/controllers, auth, permissions, external clients, async layer, shared core.
- Keep this domain-oriented unless the user asks for code-level detail.

### Deployment

Show runtime infrastructure:

- DNS/TLS, CDN/static hosting, ingress/load balancer.
- Compute platform, containers, workers, schedulers.
- Network boundaries such as VPC, public/private subnets.
- Managed databases, caches, queues, object stores, image registries, secrets/auth.
- Observability/logging.

If sources disagree, label the deployment view as "documented/assumed" and list reconciliation items.

## C4 Artifact Checklist

- Title and source notes.
- One Mermaid block per view.
- Short legend.
- Explicit uncertainty notes.
- Cross-links to narrative docs if arc42 also exists.
